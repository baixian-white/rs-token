## 1. 前置知识 · 数字星座点是什么

数字调制就是把比特流变成无线电波。每种调制方式都对应一张"星座图":

| 调制方式 | 星座点数 | 1 个符号传几 bit |
|---|:-:|:-:|
| BPSK | 2 | log₂ 2 = 1 |
| QPSK | 4 | log₂ 4 = 2 |
| 16-QAM | 16 | log₂ 16 = 4 |
| **64-QAM** | **64** | **log₂ 64 = 6** |
| 256-QAM | 256 | log₂ 256 = 8 |

每个**星座点**对应一段无线电波形。发射端把比特流按"每 N bit 一组"分块, 每组**索引**一个星座点, 把这个点的复数坐标 (I, Q) 调制到载波上发出去。

**硬性规则**: 每发射一个 64-QAM 符号, 必须凑齐 6 bit, 才能选一个星座点扔出去。

## 2. 方法 A · 普通 VQ + 调制 (痛苦)

假设训了一个 VQ-VAE, 单个大 codebook 有 1024 个 codeword:

```
codebook:
  e_0   = [64 维浮点向量]
  e_1   = [64 维浮点向量]
  ...
  e_1023= [64 维浮点向量]
```

编码 1 个特征 z: 找最近的 codeword → 输出 1 个索引 i ∈ {0..1023}, 占 **10 bit** (log₂ 1024)。

要走 64-QAM 信道, 得这么干:

```
索引 i = 537  →  二进制 1000011001  (10 bit)

切成 6 bit 一组:
  100001 1001
   ↑      ↑
  正常   只有 4 bit, 不够 6 bit !

解决方案 (都不优雅):
  1. padding 2 个 0 → 引入无意义 bit
  2. 等下一个 z 的索引来拼起来 → 多个 z 耦合
  3. 套 LDPC 信道编码再切 → 更复杂
```

**痛点**: 10 bit 不是 6 bit 的整数倍, 怎么切都难看。codebook 大小是按"重建质量"挑的, 调制方式是按"信道带宽"挑的, **两者没设计协同**。

---

## 3. 方法 B · MOC 的设计 (干净)

MOC 反过来想: **既然最终要走 64-QAM, 那就让 codebook 大小天然适配 6 bit 的整数倍**。

### 第 1 步 · 拆 codebook

不建 1 个 1024 大码本, 而是建 **4 个超小码本** (head), 每个只有 **8 个 codeword**:

```
Head 1                  Head 2                  Head 3                  Head 4
┌──────────┐            ┌──────────┐            ┌──────────┐            ┌──────────┐
│ e1_0 [16维]│            │ e2_0 [16维]│            │ e3_0 [16维]│            │ e4_0 [16维]│
│ e1_1     │            │ e2_1     │            │ e3_1     │            │ e4_1     │
│  ...     │            │  ...     │            │  ...     │            │  ...     │
│ e1_7     │            │ e2_7     │            │ e3_7     │            │ e4_7     │
└──────────┘            └──────────┘            └──────────┘            └──────────┘
  (8 个)                  (8 个)                  (8 个)                  (8 个)
```

每个 codeword **只有 16 维** (原来 64 维平分给 4 个 head)。

### 第 2 步 · 编码 z

输入特征 z (64 维) 先**切成 4 段**, 每段 16 维:

```
z = [ z_part1 (16维) | z_part2 (16维) | z_part3 (16维) | z_part4 (16维) ]
```

每段独立去对应 head 找最近 codeword:

```
z_part1 → Head 1 → 索引 a = 5  (0~7, 占 3 bit)
z_part2 → Head 2 → 索引 b = 2
z_part3 → Head 3 → 索引 c = 7
z_part4 → Head 4 → 索引 d = 1
```

**结果**: 1 个 z → 4 个索引 (5, 2, 7, 1) → 共 **12 bit** (4 × 3 = 12)。

### 第 3 步 · 喂 64-QAM (关键)

每个索引 3 bit, **两两组合正好 6 bit**, 干净对齐 64-QAM:

```
第 1 个 64-QAM 符号:
  a (3 bit) + b (3 bit) = 6 bit
  101 + 010 = 101010 = 42  →  第 42 号星座点  →  一个无线电波形

第 2 个 64-QAM 符号:
  c (3 bit) + d (3 bit) = 6 bit
  111 + 001 = 111001 = 57  →  第 57 号星座点  →  另一个无线电波形
```

→ **1 个 z 干净地变成 2 个 64-QAM 符号**, 不切片不 padding不 LDPC, 一步到位。

## 4. "8" 和 "4" 这两个数字的来源

不是拍脑袋, 是**反推自调制方式**:

### "N = 8" 的来源

```
目标: 64-QAM, 1 个符号传 6 bit
要"两个 head 拼起来 = 6 bit":
  6 bit = X bit + X bit  →  X = 3 bit
  →  每个 head = 2³ = 8 个 codeword
  →  N = 8
```

### "P = 4" 的来源

P 是工程选择, 决定"一次编码喂几个 QAM 符号":

| P | 一次编码 | 喂几个 64-QAM 符号 |
|:-:|:-:|:-:|
| 2 | 6 bit | 1 个 (信息量太少) |
| **4** | **12 bit** | **2 个 (折中, 论文取这个)** |
| 8 | 24 bit | 4 个 (codebook 太多) |

### 推广到其他调制方式

| 调制 | 1 符号 bit | N (head 大小) | 备注 |
|---|:-:|:-:|---|
| BPSK | 1 | N=2 | 1 个 head 1 个符号 |
| QPSK | 2 | N=4 或 N=2 双 head | |
| 16-QAM | 4 | N=4 双 head 拼 (2² × 2²) | |
| **64-QAM** | **6** | **N=8 双 head 拼 (2³ × 2³)** | **论文选这个** |
| 256-QAM | 8 | N=16 双 head 拼 (2⁴ × 2⁴) | |

→ **N 的选择由星座点数反推, 这就是"对齐"的工程含义**。

## 5. 方法 A vs 方法 B · 一张表对照

| 项 | 方法 A · 普通 VQ | 方法 B · MOC |
|---|---|---|
| codebook 数 | 1 | **4** |
| 每个 codebook 大小 | 1024 | **8** |
| 每个 codeword 几维 | 64 | 16 |
| 编码 1 个 z 输出 | 1 个索引 | **4 个索引** |
| 总输出 bit | 10 | 12 |
| 喂 64-QAM 怎么做 | 切片 + padding 别扭 | **3+3 拼成 6, 干净** |
| 总组合空间 | 1024 | 8⁴ = 4096 |
| 设计思路 | 先选 codebook, 再凑调制 | **先看调制要几 bit, 反推 codebook** |

## 6. 校准两个常见误解

### 误解 1 · "1024 拆成 4 个 256"

❌ 错。如果每个小 codebook 256, 总组合 256⁴ = 42 亿, 远超原来 1024。

✅ 实际是拆成 **4 个 8** (每个特别小), 总组合 8⁴ = 4096 (略大于 1024)。

**为什么是 8 而不是 256**? 因为目标是"每 head 3 bit", 不是"每 head 8 bit"。3 bit 是 64-QAM 半个符号的容量, 这是反推出来的。

### 误解 2 · "256 需要 8 bit"

✅ 算法对: log₂ 256 = 8, 但**和 MOC 无关**。MOC 选的是 8 个 codeword, 不是 256 个。

## 7. MOC 与本研究的核心区别

理解了 MOC 之后, 看清差异化:

```
传输链路:    语义提取   →   codebook 量化   →   调制传输   →   重建

MOC 的发力点:                                  ↑↑↑
                                            "对齐星座"
                                          (调制层工程)

本研究发力点:    ↑↑↑           ↑↑↑
            "RemoteCLIP        "L0 蒸馏后
              教师"           承载地物身份"
            (语义层科学)    (codebook 设计)
```

### 三个具体维度的差别

#### 维度 1 · codebook 拿来干什么 (最本质)

**MOC**: codebook 是"为了适配 64-QAM 几何而设计的量化向量"。4 阶 RVQ **只是精度递进** (先粗后细), 第一层不承载特殊语义。

**本研究**: codebook 是"为了承载遥感地物语义而设计的量化向量"。L0 通过 **RemoteCLIP 蒸馏专门承载地物身份**, L1-L3 才承载像素细节。L0 和 L1-L3 **在功能上分工**, 不只是精度递进。

#### 维度 2 · 损失函数的"语义"在哪一步

**MOC**: `L = L_recon + 0.1 × L_VGG`
- VGG19 只是个**感知损失提取器**
- VGG 没参与 codebook 的设计, 只是损失函数一部分

**本研究**: `L = L_recon + L_VGG/LPIPS + λ × L_distill_RemoteCLIP_to_L0`
- RemoteCLIP 是**专门把语义灌到 L0 codebook 的教师**
- 它**直接塑造 L0 codeword 分布**, 让它们对齐 RemoteCLIP 的特征空间
- 这是**功能性蒸馏**, 不是感知损失

#### 维度 3 · 信道极差时 (降级传输) 发生什么

**MOC**: 4 阶 RVQ 都掉了 → 重建质量塌掉。它**没有**"只传 L0 仍然可用"的设计, 因为 L0 没承载特殊任务身份。

**本研究**: 信道极差时**只传 L0** (k=1) → 重建质量明显下降, **但分类准确率仍然保住** —— 因为 L0 承载的就是地物身份, 后端任务 (分类 / 变化检测 / 灾害监测) 需要的语义还在。这就是 Stage 4 实验已验证的"k=1 ≈ k=4 在 SNR ≥ 0 dB"结论。

### 一张总差异表

| | MOC-RVQ | 本研究 |
|---|---|---|
| **创新层次** | 调制层 (工程) | 语义层 (科学) |
| **核心机制** | P × N → QAM 对齐 | RemoteCLIP → L0 蒸馏 |
| **第一层是什么** | 粗精度版本 | **语义身份层** |
| **应用域** | 自然图 (DIV2K / FFHQ) | 遥感卫星下行 (AID) |
| **下游任务** | 重建质量 (PSNR / LPIPS) | **任务保真 (分类准确率)** |
| **降级传输** | 没设计 | **k=1 单层仍保任务** |
| **信道仿真** | AWGN + 多档 QAM | AWGN + BPSK + SNR 扫描 |

## 8. 论文里怎么写差异化

**给 reviewer 看的一句话** (英文草稿):

> *MOC-RVQ optimizes the "transport layer" (how to align codebook indices with modulation constellations), while our work optimizes the "semantic layer" (what each codebook level should encode). The two are orthogonal contributions — our framework could in principle adopt MOC-RVQ's QAM alignment in addition to its semantic distillation.*

**关键姿态**: 你和 MOC-RVQ **不是竞争, 是正交**, 理论上你的方法可以**叠加**它的 QAM 对齐设计 (虽然实际你只做了更简单的 BPSK)。这是一个**既承认它的工作, 又划清贡献边界**的友好且有力的姿态。

---

## 9. 一句话回顾

**MOC-RVQ 的核心创新是"反推式 codebook 设计"** —— 不是先有大 codebook 再凑调制, 而是先看目标调制要几 bit, 再反推 codebook 必须设计成"3 bit/head, 4 head 并行"。这是个调制层的优雅工程优化, **解决"如何传"的问题**, 不解决"传什么"的问题。

**本研究的核心创新是"功能性蒸馏 codebook 设计"** —— 让 RVQ 第一层通过 RemoteCLIP 蒸馏专门承载遥感地物身份, 让信道极差时只传 L0 仍能保住任务。这是个语义层的科学贡献, **解决"传什么"的问题**, 不和 MOC-RVQ 竞争, 在另一个维度上展开。

→ **两件事可以叠加, 不冲突。**
